# 航空嵌入式 Markdown 文档集合规范

## 目标

本规范定义一套面向航空嵌入式软件的 DO-178C 风格 Markdown 文档集合。默认场景是 Linux 嵌入式软件，后续可扩展到 RTOS 或裸机平台。模板用于工程开发、审查、追踪和验证证据准备，不替代正式认证计划或适航批准材料。

## 默认约定

- 文档语言：中文为主，保留 DO-178C、DAL、HLR、LLR、MC/DC、traceability 等英文术语。
- 默认等级：DAL B/C 风格。
- 默认平台：Linux 嵌入式；涉及调度、并发、时限、内存、硬件耦合时必须填写 RTOS/裸机迁移影响。
- 文档状态：Draft、In Review、Approved、Baseline、Obsolete。
- 验证方法：Review、Analysis、Test、Inspection、Demonstration。

## ID 规则

| 对象 | ID 格式 | 示例 |
|---|---|---|
| 系统需求 | `SYS-REQ-0001` | `SYS-REQ-0007` |
| 高层需求 | `HLR-0001` | `HLR-0012` |
| 低层需求 | `LLR-0001` | `LLR-0034` |
| 设计元素 | `DES-0001` | `DES-0009` |
| 代码单元 | `CODE-<module>-0001` | `CODE-nav-0001` |
| 单元测试 | `UT-0001` | `UT-0020` |
| 集成测试 | `IT-0001` | `IT-0005` |
| 问题/偏差 | `ISSUE-0001` | `ISSUE-0003` |
| 变更 | `CHG-0001` | `CHG-0004` |

## 需求质量规则

- 单条需求只表达一个可验证行为或约束。
- 需求必须有来源、理由、验收准则、验证方法和追踪链接。
- 系统需求描述外部可见能力或约束；HLR 描述软件层面的功能、接口、时序和错误处理；LLR 描述模块可实现和可单元测试的行为。
- 避免使用“尽量”“快速”“合适”“稳定”等不可验证词，除非同时给出度量。
- 不在 HLR 中写入过细代码结构，不在 LLR 中丢失上层意图。

## 追踪链

最小追踪链为：

`SYS-REQ -> HLR -> LLR -> DES/CODE -> UT/IT -> Verification Summary`

允许多对多关系，但每个链接必须能解释覆盖意图。追踪矩阵中缺失、部分覆盖、废弃、延期验证都要显式标注。

## Linux 与 RTOS 字段

Linux 文档应关注：

- 进程、线程、调度策略、CPU 亲和性、优先级。
- IPC、设备节点、sysfs/procfs、netlink、socket、共享内存。
- 驱动接口、DMA/中断暴露边界、错误码、日志和恢复策略。
- 目标机、内核版本、库版本、构建配置和启动顺序。

RTOS/裸机迁移预留应关注：

- 任务、ISR、调度周期、优先级反转、互斥和队列。
- 栈/堆、水位、静态分配、最坏执行时间、启动时序。
- 硬件寄存器、DMA、中断屏蔽、确定性和资源上限。

## 审查和证据

- DAL B/C 风格默认要求需求审查、设计审查、代码审查、测试审查和追踪审查。
- 对独立性有要求时，作者和审查人不能相同。
- 任何未通过项必须关联 `ISSUE-*`，并记录关闭证据。
- 验证结果必须包含环境、版本、输入、预期结果、实际结果和结论。
