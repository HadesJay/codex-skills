## 1. 通用规则

| ID | 规则 | 理由 | 例外审批 |
|---|---|---|---|
| CS-GEN-0001 | C 代码默认使用 C99，不使用编译器私有扩展，除非记录理由、编译器版本和回退方案。 | 保持可移植性和可审查性。 | 需要 |
| CS-GEN-0002 | 所有外部接口必须有明确错误码、错误传播规则和恢复策略。 | 支持可验证的故障处理。 | 需要 |
| CS-GEN-0003 | 禁止未初始化变量、未检查返回值和隐式资源泄漏。 | 降低运行时不确定性。 | 不允许 |
| CS-GEN-0004 | 新增并发共享数据必须说明锁、原子操作或无锁依据。 | 防止竞态和优先级问题。 | 需要 |
| CS-GEN-0005 | 所有 deviation 必须记录规则 ID、原因、风险、替代措施和审批人。 | 保证规则偏离可追踪。 | 不允许无记录偏离 |

## 2. 命名和格式规则

| ID | 规则 | 示例/说明 | 例外审批 |
|---|---|---|---|
| CS-FMT-0001 | 文件名、目录名、函数名、变量名使用 `snake_case`。 | `flight_data_parser.c`、`parse_frame()` | 需要 |
| CS-FMT-0002 | 宏和枚举常量使用 `UPPER_SNAKE_CASE`。 | `MAX_FRAME_LEN`、`STATE_INIT` | 需要 |
| CS-FMT-0003 | 类型名使用项目统一后缀，结构体类型建议 `xxx_t`。 | `sensor_sample_t` | 需要 |
| CS-FMT-0004 | 私有函数使用 `static`，可加模块前缀避免名称冲突。 | `static int nav_parse_frame(...)` | 不允许无理由外部暴露 |
| CS-FMT-0005 | 缩进使用 4 个空格，不使用 Tab；行宽建议不超过 100 列。 | 便于审查和 diff。 | 需要 |
| CS-FMT-0006 | 花括号风格在项目内保持一致；所有 `if/for/while/do` 块必须使用花括号。 | 禁止单行无括号分支。 | 不允许 |

## 3. C99 语言和类型规则

| ID | 规则 | 理由 | 例外审批 |
|---|---|---|---|
| CS-C99-0001 | 使用 `stdint.h` 固定宽度整数表达硬件、协议和持久化数据。 | 避免平台位宽歧义。 | 需要 |
| CS-C99-0002 | 指针参数必须说明所有权、生命周期和是否允许 `NULL`。 | 防止悬空指针和空指针错误。 | 不允许无说明 |
| CS-C99-0003 | 禁止隐式窄化转换；有符号和无符号混用必须显式处理边界。 | 防止溢出和比较错误。 | 需要 |
| CS-C99-0004 | 禁止魔法数字；协议值、寄存器位、时限和容量必须用命名常量。 | 提升追踪和审查效率。 | 需要 |
| CS-C99-0005 | 全局变量必须最小化；可变全局状态必须说明初始化顺序、并发保护和测试复位方式。 | 降低隐藏耦合。 | 需要 |
| CS-C99-0006 | `volatile` 只用于硬件寄存器、ISR 共享对象或明确的内存映射 I/O，不得作为并发同步替代。 | 避免错误同步假设。 | 不允许误用 |
| CS-C99-0007 | 动态内存默认禁止用于安全关键路径；如使用，必须证明上限、失败处理和释放路径。 | 保持确定性。 | 需要 |

## 4. 函数和控制流规则

| ID | 规则 | 理由 | 例外审批 |
|---|---|---|---|
| CS-CTL-0001 | 函数只承担单一职责，建议不超过 80 行，圈复杂度目标不超过 10。 | 降低审查和测试复杂度。 | 需要 |
| CS-CTL-0002 | 公共函数必须校验输入参数并定义失败返回值。 | 防止错误扩散。 | 不允许 |
| CS-CTL-0003 | 返回值必须被检查；故意忽略时必须用项目约定宏或注释说明原因。 | 保证错误不被吞掉。 | 需要 |
| CS-CTL-0004 | `switch` 必须覆盖所有枚举值；`default` 中应记录异常处理或不可达理由。 | 防止新增状态漏处理。 | 需要 |
| CS-CTL-0005 | `goto` 仅允许用于统一资源释放路径，不允许跨越复杂控制流。 | 降低泄漏风险并保持可读性。 | 需要 |
| CS-CTL-0006 | 递归默认禁止。 | 避免栈深度不可控。 | 需要 |
| CS-CTL-0007 | 中断、信号或回调上下文中不得执行阻塞、不可重入或无界耗时操作。 | 保证时序和稳定性。 | 不允许 |

## 5. 协议与内存布局规则

| ID | 规则 | 理由 | 例外审批 |
|---|---|---|---|
| CS-MEM-0001 | 跨设备、跨进程或持久化数据必须显式约定字节序，并在序列化/反序列化处集中转换。 | 防止异构平台数据错位。 | 需要 |
| CS-MEM-0002 | 协议、寄存器和外部 ABI 结构体必须显式标注对齐与填充策略，必要时使用项目封装的 `packed` 宏。 | 避免编译器隐式重排导致的解析错误。 | 需要 |
| CS-MEM-0003 | 缓冲区操作必须传入容量并校验长度，禁止无界拷贝和裸 `strcpy/sprintf`。 | 防止越界与未定义行为。 | 不允许 |
| CS-MEM-0004 | 位域不得用于硬件寄存器或网络协议的可移植解析。 | C 标准未规定位域顺序与对齐。 | 不允许 |

## 6. Linux 嵌入式规则

| ID | 规则 | 适用范围 | 例外审批 |
|---|---|---|---|
| CS-LNX-0001 | 系统调用、pthread、文件、socket、ioctl 返回值必须检查。 | 用户态/驱动接口 | 不允许 |
| CS-LNX-0002 | 与设备节点、sysfs/procfs、netlink 交互必须记录 ABI 假设。 | 平台接口 | 需要 |
| CS-LNX-0003 | 日志不得泄露敏感数据，且不得影响实时路径。 | 运行时 | 不允许 |
| CS-LNX-0004 | 线程创建、优先级、CPU 亲和性和调度策略必须在设计或代码注释中可追踪。 | 并发/实时路径 | 需要 |
| CS-LNX-0005 | 文件描述符、内存映射、锁、线程和 socket 必须有明确释放路径。 | 资源管理 | 不允许 |

## 7. RTOS/裸机迁移规则

| ID | 规则 | 迁移关注点 | 例外审批 |
|---|---|---|---|
| CS-RTOS-0001 | 动态内存使用必须可证明有界或替换为静态分配。 | 确定性 | 需要 |
| CS-RTOS-0002 | 任务、ISR、队列和互斥必须记录最坏路径。 | 时序 | 需要 |
| CS-RTOS-0003 | 硬件寄存器访问必须集中封装并标注副作用。 | 可测试性 | 需要 |
| CS-RTOS-0004 | 栈使用、队列深度、定时器周期和任务优先级必须有上限依据。 | 资源上限 | 需要 |
| CS-RTOS-0005 | ISR 与任务共享数据必须使用目标平台认可的同步机制。 | 并发正确性 | 不允许 |

## 8. 静态分析与工具链规则

| ID | 规则 | 理由 | 例外审批 |
|---|---|---|---|
| CS-TOOL-0001 | 编译器告警视为错误（`-Wall -Wextra -Werror` 或等价配置），新增豁免必须记录范围与时限。 | 防止告警噪音掩盖缺陷。 | 需要 |
| CS-TOOL-0002 | MISRA C 子集（具体版本与豁免清单见元表）必须在 CI 中强制执行，违规须挂接 deviation 流程。 | 与 DO-178C/适航审查对齐。 | 需要 |
| CS-TOOL-0003 | 编译器、静态分析工具及其版本必须冻结并写入基线，与项目计划一致。 | 保证证据可重现。 | 需要 |
| CS-TOOL-0004 | 第三方/开源代码必须标注来源、版本、许可证和裁剪范围，并独立追踪其偏差。 | 隔离非项目代码风险。 | 需要 |

## 9. 规则追溯与引用

- 本文件所有规则 ID（`CS-*`）应在以下位置可被反向引用：
  - 静态分析配置（如 `.clang-tidy`、MISRA 检查器规则映射表）。
  - 逐项编码审查 [13_code_review_against_standard.md](13_code_review_against_standard.md)（逐条 `CS-*` 结论与证据）；汇总与关单写入 [07_code_review.md](07_code_review.md)。
  - LLR [04_low_level_requirements.md](04_low_level_requirements.md) 中与编码相关的派生需求。
  - Deviation 登记表（项目级 waiver list，未提供时填 TBD）。
- 新增、修改、废弃规则必须更新版本与基线字段，并在 [12_change_impact_analysis.md](12_change_impact_analysis.md) 中说明影响范围。

## 10. 审查清单

- 是否满足 C99、命名、格式、接口、错误处理、资源释放和并发规则。
- 是否存在未审批的编译器扩展、动态内存、递归、隐式转换或全局可变状态。
- 是否存在未约束的字节序/对齐/位域使用，或裸长度无校验的缓冲区操作。
- 是否存在不可验证假设或隐藏平台依赖。
- 是否所有违规已挂接到 deviation 流程或形成新的需求/设计/测试追踪项。
